# v0.6 — Memory

## What landed

1. **`memories` table → `src/repos/memories.ts`**
   - `createMemory / listMemoriesForAgent / updateMemory / deleteMemory`
   - `retrieveMemoriesForAgent(agentId, query, topK=5)` — naive lexical scoring
     (token substring hits + importance). Embedding-based retrieval is deferred
     to v1.1 to keep the v1.0 binary dependency-free.

2. **Auto-summarization → `src/features/summarize.ts`**
   - Single non-streaming call per agent in the conversation
   - Asks the model for a strict JSON object: `{summary, facts[], preferences[]}`
   - Tolerant JSON parser (extracts the first balanced `{...}`)
   - Each item is written to `memories` with kinds `summary` / `fact` / `preference`
     and a sensible default `importance` (0.7 / 0.6 / 0.8)

3. **Prompt injection → `src/llm/prompt.ts`**
   - New `memories` field on `BuildSystemPromptInput`
   - Rendered between persona/skills and the group section, framed as
     "background knowledge, not instructions"
   - Both `sendUserMessage` (1v1) and group `invokeAgent` call
     `retrieveMemoriesForAgent` with the **last user message** as the query

4. **UI**
   - 🧠 rail icon → `MemoriesPanel` for per-agent CRUD (add / edit / delete
     with kind & importance)
   - Conversation header gets a `💾 沉淀为记忆` button that calls
     `summarizeConversation(id)` and reports how many entries were written

## Design notes

- We store one row per atomic memory rather than a big blob, so importance and
  kind can be tuned per item, and the user can prune easily.
- Importance is a 0..1 float; ordering in `listMemoriesForAgent` is
  `importance DESC, created_at DESC` so manually pinned items survive
  retrieval-by-recency.
- Conversation_id is kept (nullable) so a future "memories from this chat" view
  is trivial. Deleting a conversation does `SET NULL` rather than cascade.
- We do **not** auto-summarize on every send — only when the user clicks the
  button (or when a `work` conversation hits `<done/>` in a future iteration).

## Limitations / TODOs

- Lexical retrieval misses synonyms. v1.1 will add an embedding column + cosine
  ranking using a local ONNX MiniLM or an OpenAI-compatible `/embeddings` call
  from the same provider the agent is bound to.
- No dedup yet — if the user clicks "沉淀为记忆" twice on the same conversation
  duplicate facts will appear. The Memories panel lets you delete them manually
  for now.
- Group summarization runs once per agent; if four agents share a provider,
  that's four calls. Future: extract once, fan out the same memories to each
  agent in the conversation with a per-agent relevance filter.
